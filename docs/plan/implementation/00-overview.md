## 2. Current state (verified against the repo, June 2026)

**Backend**
- 10 axis FastAPI services in `backend/services/*`. Each exposes:
  - `POST /diagnose` — **works** (evaluate mode): returns `score`, `blockers`,
    `justification`, `due_diligence`, etc.
  - `POST /execute` — **stub**: returns `{"mode":"execute","status":"not_implemented"}`.
  - `POST /metric_update` — daemon hook.
- Orchestrator (`backend/orchestrator/app`): `state_router` (project lifecycle),
  `intake_router` (stateless intake), `diagnostic_router` (`run-diagnostic`
  fan-out + roadmap), `history_router`, `tools_router`, `redis_consumer` → SSE.
- `axis_registry.py` defines ports, the metric→axis routing table, and the
  3-wave diagnostic dependency order. **Axis 10 is `gtm`, not `roadmap`.**
- Roadmap is generated by `diagnostic_router._call_roadmap` via `backend/rag`
  (RAG over Tunisian resources), not by a standalone axis service.

**Database** (`db/migrations/001..005`)
- `profiles` (doubles as the project row: `id, name, state(NEW|EXISTING), sector,
  language, profile JSONB`), `diagnostic_history`, `alerts`, `score_snapshots`,
  `roadmap_versions`, `tool_integrations`, `roadmap_action_status`.

**Frontend** (`frontend/src`)
- `App.tsx` (635 lines) drives routing between landing / intake / creation /
  dashboard. `store.ts` (Zustand), `api.ts`, `types.ts`.
- `CreationFlow.tsx` **fakes** creation: it runs `/run-diagnostic` and builds
  review cards from `axis_outputs` because `/execute` is stubbed.
- Single-project dashboard; `RoadmapTimeline` has working action checkboxes;
  `mon-parcours` has history + completed actions.
- Daemon alerts → SSE → `AlertFeed`.

## 4. Architectural decisions (resolve before coding)

These choices are referenced throughout the workstream files. Defaults are
proposed; flag any you want to change.

1. **`projects` vs `profiles`.** Do **not** add a new `projects` table. Extend
   the existing `profiles` row (it already is the project) so we keep working
   intake/diagnostic machinery. The design doc's `projects.mode`
   (creation/diagnosis) maps onto the existing `state` column
   (`NEW`→creation, `EXISTING`→diagnosis). See `01-data-model.md §1`.

2. **Generate endpoint.** Replace the `/execute` stub with a real
   `POST /generate` on every axis (keep `/diagnose` for evaluate). The
   orchestrator picks the endpoint from the run `mode`. See `02-axis-dual-mode.md`.

3. **Axis 10.** Treat **Roadmap** as the orchestrator's RAG roadmap generator
   (not a network service). Keep `operations` as a first-class scoring axis and
   retire `gtm` as a separate axis, folding GTM concerns into `marketing`/`sales`
   per the design doc's 10-axis list. See `02-axis-dual-mode.md §4` and `09 §Risks`.

4. **Dependency graph storage.** Encode the graph in code
   (`axis_registry.py`) as the source of truth and *optionally* mirror it to a
   `dependency_graph` table for traceability/versioning. Re-running is a code
   path, not a DB query. See `05-dependency-engine.md`.

5. **Re-run UX.** Every auto re-run (chat/tool/daemon/manual edit) produces
   **proposals the user reviews** (creation mode) or **recomputed scores**
   (diagnosis mode) — never silent overwrites. Each is logged to `events`.

6. **LLM + RAG stay local** (Ollama `llama3.1:8b`, `nomic-embed-text`, Qdrant)
   per project principle "100% local". No cloud calls added.

## 5. Cross-cutting conventions

- **Due diligence:** every axis `/diagnose` keeps returning its `due_diligence`
  block (existing project convention). New `/generate` output does **not** score,
  so it carries no DD block.
- **Slugs:** canonical axis slugs stay `ideation, market, product, brand,
  business-model, legal, operations, marketing, sales` + the virtual `roadmap`.
- **SSE:** all long-running fan-outs (generate loop, re-runs, diagnostic) stream
  progress on the existing SSE channel (`sse.py` / `redis_consumer.py`).
- **i18n:** every new UI string lands in `frontend/src/locales/{fr,en,ar}.json`
  at parity (keys currently at 140 each). RTL already handled.
- **Migrations:** additive only, `006`+; never edit shipped migrations.

## 6. Suggested phase order (detail in `09`)

1. **Phase A — Foundations:** data model (`01`), generate mode (`02`),
   dependency engine (`05`). Unblocks everything else.
2. **Phase B — Flows:** creation loop + PDF export (`03`), diagnosis + PDF
   import + debate (`04`), multi-project UI (`08`).
3. **Phase C — Liveness:** continuous updates + event feed (`06`), tool signals
   (`06`/`10`), adaptive roadmap + KB versioning (`07`).
4. **Phase D — Polish:** "What's new?", diff viewer, celebration/regen UX.

The hackathon **demo cut-line** is end of Phase B + the daemon-event card and
progress-aware roadmap from Phase C (see `09`).

### Post-D phases (build order E → F → G)

   self-hosted SearXNG live web search, evidence injected once in the generation
   runner, citations surfaced in the UI. *(Implemented — see the master plan and
   `scripts/generate_kb_axis.py`.)*
6. **Phase F — Real daemon + companion lifetime:** `11-daemon-liveness-and-companion.md`.
7. **Phase G — No-code bidirectional integrations:** `12-composio-integrations.md`.
