# 07 ‚Äî Roadmap Engine: Live, Adaptive, Versioned KB

> Implements new-logic.md ¬ß6. The roadmap (axis 10, virtual) is RAG-generated,
> progress-aware, score-reactive, and records which KB version produced it.
> Builds on `backend/rag`, `diagnostic_router._call_roadmap`, and the existing
> `roadmap_action_status` completion tracking.

---

## 1. Current state

- Roadmap is generated by `diagnostic_router._call_roadmap` via RAG over
  Tunisian resources; nested shape `roadmap.roadmap.{immediate, short_term,
  medium_term}`, each action `{action, rationale, resource:{title,url}}`.
- `roadmap_versions` stores versions; `roadmap_action_status` tracks per-action
  completion (`action_key = ${horizon}:${djb2(text)}`); `RoadmapTimeline` has
  working checkboxes; `CompletedActions` in Mon Parcours.

Gaps: progress-aware regeneration (¬ß6.2), score-driven re-prioritisation (¬ß6.3),
KB versioning/provenance (¬ß6.1), stale-flag + approve-to-regen cadence (¬ß6.4).

---

## 2. Generation cadence (¬ß6.4)

Three triggers, **never on a timer**:
- **On-demand** ‚Äî "Re-generate roadmap" button ‚Üí `POST /project/{id}/roadmap/regenerate`.
- **After significant update** ‚Äî when `dependency.rerun` touches axes whose
  output materially changed, mark the live `roadmap_versions.stale=true`
  (`01 ¬ß6`) and surface a "roadmap may be out of date ‚Äî regenerate?" prompt.
  **Not automatic** ‚Äî user approves.
- **On completion milestones** ‚Äî see ¬ß3.

Each regeneration inserts a new `roadmap_versions` row with `kb_version`,
`trigger`, and clears `stale`.

## 3. Progress-aware roadmap (¬ß6.2)

When the user checks an action (`POST /project/{id}/roadmap/action`, exists):
- Mark complete (exists) + UI: green check, strikethrough, opacity, counter,
  celebration animation (Moufida reacts via companion state event), move to
  Completed log (Mon Parcours, exists).
- **New:** when a horizon's actions are *all* complete, generate next-horizon
  actions from what was accomplished:
  - `generate_next_actions(project_id, horizon)` ‚Äî RAG call seeded with the
    completed actions + current plan/scores ‚Üí appends new actions to the next
    horizon, persists a new roadmap version (`trigger='completion'`).
  - Collapse the completed horizon with a badge; shift the next horizon up as
    "current focus"; Moufida announces progress.

`POST /project/{id}/roadmap/advance` body `{horizon}` encapsulates this; the UI
calls it when it detects a horizon fully checked.

## 4. Score-driven re-prioritisation (¬ß6.3)

On a diagnostic re-run or significant score change, compare per-axis scores to
the previous snapshot (`score_snapshots`, exists):
- Axis score **drops ‚â• 1.0** ‚Üí bump its related actions up one horizon
  (medium‚Üíshort, short‚Üíimmediate) and propose a new immediate action.
- Axis score **improves ‚â• 1.0** ‚Üí archive its completed actions, propose more
  advanced ones.
- Emit a notification badge/event: "3 actions re-prioritised due to Market
  score change" (writes an `events` row, `source` per trigger).

Implement as `reprioritize(project_id, score_deltas)` in the roadmap module,
called from the diagnosis re-run path (`04 ¬ß6`) and the dependency scheduler
(`05 ¬ß4`) when scores move. Map axis‚Üíaction via the action's owning axis (store
`axis_slug` on each generated action so re-prioritisation can find them).

## 5. Evolving, versioned RAG KB (¬ß6.1, ¬ß8)

- Backing store: `knowledge_base` table (`01 ¬ß5`) for catalogue + `kb_version`;
  **Qdrant** for vectors keyed by `knowledge_base.id`.
- **Sources** feed in:
  - *Initial seed* ‚Äî existing curated Tunisian resources (current RAG corpus) ‚Üí
    `source='seed'`, global (`project_id NULL`).
  - *Tool integrations* ‚Äî structured tool data becomes searchable context
    (`source='tool'`, project-scoped).
  - *User uploads* ‚Äî diagnosis PDFs (`04 ¬ß2`) ‚Üí `source='upload'`, project-scoped;
    optional anonymised promotion to global KB (with permission flag).
  - *Manual additions* ‚Äî `POST /project/{id}/kb` (notes/links) ‚Üí `source='manual'`.
  - *External feeds* ‚Äî daemon-pulled regulatory/market feeds ‚Üí `source='feed'`.
- **Versioning:** any KB mutation bumps a monotonic `kb_version` (global +
  per-project). Each roadmap generation records the `kb_version` it retrieved
  over (`roadmap_versions.kb_version`) so the UI shows "based on these sources".
- Retrieval at roadmap-gen time = global seed ‚à™ project-scoped rows for the
  current `kb_version`.

## 6. API surface

| Method & path | Purpose |
|---------------|---------|
| `POST /project/{id}/roadmap/regenerate` | on-demand regen (records kb_version, trigger) |
| `POST /project/{id}/roadmap/advance` | next-horizon generation on full completion |
| `POST /project/{id}/kb` | manual KB addition (bumps kb_version) |
| `GET  /project/{id}/roadmap/provenance` | KB sources/version behind the live roadmap |

Existing: `GET /roadmap`, `POST /roadmap/action`, `GET /roadmap/actions`.
