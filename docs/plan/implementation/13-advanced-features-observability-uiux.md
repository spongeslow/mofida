# Phase H: Advanced Analytical Features, Observability Platform & UI/UX Overhaul

> **Scope:** Four parallel workstreams — H1 Investor Pitch Simulator, H2 Pivot Scenario
> Planner, H3 Customer Persona Simulator, H4 Web Admin / Observability Platform — plus a
> cross-cutting H5 UI/UX layer that applies to all of them and to existing screens.
>
> **Design principle for H1–H3:** Every generated output must be evidence-grounded and
> reasoning-traceable. No hallucinated claims. Every claim links to a source: an axis
> output field, a KB document, a competitor snapshot, or a daemon signal event. The UI
> surfaces this chain explicitly so judges (and founders) can follow the logic.

---

## H1 — Investor Pitch Simulator

### What it solves

Founders in Tunisia often fail investor meetings not because their startup is weak but
because they have never rehearsed the hard questions. This simulator gives them a
controlled, private, AI-driven investor persona to spar against — grounded entirely in
**their own diagnostic data**, not generic advice.

### Evidence-grounding contract

The simulator may only challenge the founder on claims it can verify or refute from:

| Source | Example claim | Challenge it unlocks |
|---|---|---|
| Axis output fields | `market.score = 1.8/5` | "Your market validation score is critically low — who are your first 10 customers?" |
| Blockers list | `critical: no_ip_protection` | "You have no IP protection on record. Why shouldn't a competitor clone this tomorrow?" |
| Competitor board | `competitor_snapshots[n].pricing` | "Competitor X already offers this at €29/month. Why would someone pay you €49?" |
| Opportunity radar | `opportunities` rows | "You missed the ANPR R&D grant deadline last month. What's your funding runway?" |
| KB retrieval | RAG chunks tagged with source | "Most agri-food B2B SaaS plays in MENA fail at distribution, not product. Address that." |
| Event feed | Recent daemon signals | "Your legal watcher flagged a regulatory change 3 weeks ago — have you acted on it?" |

Every investor question generated by Ollama is **annotated with the source** before being
sent to the frontend. The UI renders the source as a collapsible trace panel.

### Backend — new endpoints

```
POST /api/v1/project/{id}/pitch/start
    body: { investor_profile: "seed_vc" | "angel" | "impact_fund" | "strategic" }
    → { session_id, opening_question, trace }

POST /api/v1/project/{id}/pitch/respond
    body: { session_id, answer: string }
    → { follow_up_question, trace, score_delta?: { axis, delta, reason } }

POST /api/v1/project/{id}/pitch/end
    body: { session_id }
    → PitchReadinessReport {
        overall_readiness: float,  // 0–100
        per_axis_readiness: Record<Axis, { score: float, gaps: string[] }>,
        hardest_questions: string[],
        evidence_used: EvidenceRef[],
        recommended_actions: string[]
      }
```

**`pitch_router.py`** — session state held in Redis (session_id → conversation history +
evidence snapshot). Each `respond` call:
1. Feeds the answer + evidence snapshot into Ollama with a strict prompt template.
2. Ollama returns `{ question, reasoning, evidence_refs[], score_impact? }` as JSON.
3. `llm_json.py` validates the schema; falls back to `parse_json` if needed.
4. Stores the exchange in `pitch_sessions` table (new migration 017).
5. Returns the question and its full trace to the frontend.

**New migration `017_pitch_sessions.sql`:**
```sql
CREATE TABLE pitch_sessions (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id  UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
    investor_profile TEXT NOT NULL,
    exchanges   JSONB NOT NULL DEFAULT '[]',
    readiness   JSONB,
    started_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    ended_at    TIMESTAMPTZ
);
```

## H2 — Pivot Scenario Planner

### What it solves

Before committing to a strategy change, a founder needs to see its projected impact on
all 9 axes with the underlying reasoning, not just a gut feeling. The Planner generates
up to 3 side-by-side pivot scenarios, each with projected score changes and cited KB
evidence.

### Evidence-grounding contract

A scenario is a set of **parameter overrides** applied to the founder's current profile.
The projection engine:

1. Computes a diff between current profile fields and the scenario overrides.
2. For each changed field, queries the KB (`/rag/query`) with a structured prompt:
   `"In sector X, when a startup shifts from Y to Z, how does [axis] typically change?"`
3. Collects RAG chunks (with source doc IDs) and feeds them to Ollama alongside the
   current axis output for that axis.
4. Ollama returns `{ projected_score, delta, confidence, reasoning, sources[] }`.
5. Confidence intervals are shown (not just point estimates) to avoid false precision.

The UI never shows a projected score without a confidence bar and its source list.

### Backend — new endpoints

```
POST /api/v1/project/{id}/scenario/project
    body: {
      label: string,
      overrides: Partial<ProfileFields>  // e.g. { target_segment: "B2B", pricing_model: "SaaS" }
    }
    → ScenarioProjection {
        label,
        overrides,
        axis_projections: Record<Axis, {
          current_score: float,
          projected_score: float,
          delta: float,
          confidence: "high" | "medium" | "low",
          reasoning: string,
          sources: EvidenceRef[]
        }>,
        overall_delta: float,
        generated_at: timestamp
      }

GET  /api/v1/project/{id}/scenarios
    → ScenarioProjection[]   // last 10, ordered by generated_at DESC

POST /api/v1/project/{id}/scenario/{label}/adopt
    → triggers profile patch with the overrides + re-diagnostic
```

**`scenario_router.py`** — no persistent table needed (projections are ephemeral; they
live in the frontend until adopted). Each projection runs all 9 axis projections in
parallel (`asyncio.gather`). Each projection call is a RAG query + Ollama call via
`llm_json.py`; results are tagged with a `projection_id` UUID for traceability.

## H3 — Customer Persona Simulator

### What it solves

A founder's biggest blind spot is assuming they know who their customer is. This
simulator generates 3–5 realistic personas derived from the actual diagnostic data and
KB sector knowledge, then lets the founder have a realistic conversation with each —
hearing objections, FAQs, and buying triggers in the customer's voice.

### Evidence-grounding contract

Each persona is constructed from:

| Data source | What it contributes |
|---|---|
| `market` axis output | Segment definition, ICP, willingness-to-pay range |
| `product` axis output | Core value proposition, known gaps |
| `brand` axis output | Positioning vs competitors |
| Competitor board | How competitor customers are positioned differently |
| KB RAG | Sector-specific customer archetypes, typical objections |
| `offer` profile field | Pricing tier, delivery model |

The persona is a structured JSON object stored in `customer_personas` table, with every
field annotated with a `source_ref`. The persona chat prompt fed to Ollama includes the
full source map so every response is grounded.

Persona responses in chat are **required** to include, in hidden metadata:
`{ claim, source_ref, confidence }` for each substantive statement — the UI renders these
as toggleable footnotes.

### Backend — new endpoints

```
POST /api/v1/project/{id}/personas/generate
    → PersonaSet { personas: Persona[], generated_at, evidence_refs[] }

GET  /api/v1/project/{id}/personas
    → Persona[]

POST /api/v1/project/{id}/persona/{persona_id}/chat
    body: { message: string, history: ChatTurn[] }
    → { reply: string, claims: Claim[], buying_signal?: string, objection?: string }

GET  /api/v1/project/{id}/persona/{persona_id}/close-strategy
    → { strategy: string, key_triggers: string[], objections_to_address: string[] }
```

## H4 — Web Admin / Observability Platform

### Purpose

A standalone browser-accessible admin panel at `http://localhost:3002` that lets judges,
evaluators, and the developer observe Moufida's internal behaviour in real time. It
demonstrates production-readiness and provides a transparent view into what would
otherwise be a black-box LLM system.

This is the **single biggest technical credibility signal** in the demo: showing request
traces, LLM prompt/response pairs, daemon activity, and KB health in a live view proves
the system is engineered, not hacked together.

### Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│  Admin frontend (Vite + React, port 3002)                       │
│  admin/  directory at repo root — standalone app               │
└───────────────────────┬─────────────────────────────────────────┘
                        │ REST + SSE
┌───────────────────────▼─────────────────────────────────────────┐
│  FastAPI /api/admin/* routes (admin_router.py)                  │
│  — protected by a simple local shared secret (ADMIN_TOKEN)      │
│  — middleware writes telemetry into 3 new PostgreSQL tables      │
└─────────────────────────────────────────────────────────────────┘
```

The admin panel is read-only and local-only. It does not change any application state.

### New telemetry infrastructure

**Middleware (`telemetry_middleware.py`):**
- Intercepts every incoming request to the orchestrator.
- Assigns `X-Request-ID` (UUID) if not present.
- After the handler completes, writes a row to `api_requests` with method, path,
  status code, duration, and the request ID.
- Attaches the request ID to a `contextvars.ContextVar` so any downstream code
  (LLM calls, axis runs) can tag their own telemetry rows.

**New migration `019_telemetry.sql`:**
```sql
CREATE TABLE api_requests (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    request_id  UUID NOT NULL,
    method      TEXT NOT NULL,
    path        TEXT NOT NULL,
    status_code INT  NOT NULL,
    duration_ms INT  NOT NULL,
    project_id  UUID REFERENCES profiles(id) ON DELETE SET NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX ON api_requests (created_at DESC);

CREATE TABLE llm_calls (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    request_id   UUID,
    axis         TEXT,
    model        TEXT NOT NULL,
    prompt_hash  TEXT NOT NULL,  -- SHA-256, not stored raw (privacy)
    prompt_preview TEXT NOT NULL, -- first 280 chars
    response_preview TEXT,        -- first 280 chars
    duration_ms  INT,
    tokens_in    INT,
    tokens_out   INT,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE daemon_activities (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id   UUID REFERENCES profiles(id) ON DELETE SET NULL,
    watcher      TEXT NOT NULL,
    activity     TEXT NOT NULL,  -- e.g. "competitor_page_changed", "grant_found"
    detail       JSONB,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX ON daemon_activities (created_at DESC);
```

`llm_json.py`'s `generate_json()` function writes a `llm_calls` row after every Ollama
call. The daemon POSTs activity rows to a new lightweight endpoint
`POST /api/v1/daemon/activity` (added to `daemon_router.py`).

### Admin router (`admin_router.py`)

All routes require `Authorization: Bearer {ADMIN_TOKEN}` header (checked against env var,
returns 401 if missing or mismatched; if `ADMIN_TOKEN` is unset, auth is disabled for
local-only convenience).

```
GET  /api/admin/health
    → {
        services: {
          postgres: { status, latency_ms },
          redis:    { status, latency_ms },
          qdrant:   { status, latency_ms, collection_count, total_vectors },
          ollama:   { status, latency_ms, loaded_models[] },
          searxng:  { status, latency_ms },
          daemon:   { alive, last_beat, focus_project_id, paused }
        },
        kb: {
          axes: Record<Axis, { doc_count, last_ingested, staleness_days }>,

        }
      }

GET  /api/admin/requests?limit=50&path_prefix=/api/v1/project
    → { rows: ApiRequest[], total: int }

GET  /api/admin/trace/{request_id}
    → {
        request: ApiRequest,
        llm_calls: LlmCall[],
        axis_outputs: AxisOutput[],    -- if it triggered any axis runs
```

## H5 — UI/UX Overhaul: Animations, Character & Themes

This section applies to all existing and new screens. It is a cross-cutting quality
layer, not a new feature. Every subsection below maps to a concrete set of CSS/code
changes.

### H5.1 — 2D Character: new states and movements

**New `CharacterState` values (extend existing type):**

| State | When | Visual description |
|---|---|---|
| `"skeptic"` | Pitch simulator session | Arms crossed, slight forward lean, darker dress tint overlay |
| `"presenting"` | Scenario results, Board Report ready | One arm raised pointing right, upright posture |
| `"pointing_left"` | Persona gallery — directing to a card | Left arm extended horizontally |
| `"surprised"` | High-impact daemon signal arrives | Arms at sides but body jolts up 3px (CSS keyframe) |
| `"worried"` | Axis score drops below 1.5 | Slight slump, frown replaces smile in SVG |
| `"wave"` | First boot, welcome screen | Existing wave from PixelMoufida — also add to SVG character |
| `"reading"` | KB staleness check running | Character looks down, slight head tilt |

**SVG costume overlays** — implemented as `<g className="mf-costume-{name}">` layers
inside `MoufidaCharacter.tsx` that are `display:none` by default and activated by the
state class, similar to how `mf-sleep-zzz` and `mf-sparkles` work today:

- `mf-costume-skeptic`: darkens dress fill from `#C96A2D` to `#8B4A1E`, adds a tiny
  briefcase in right hand (simple rectangle + handle path).
- `mf-costume-presenting`: adds a small pointer/wand prop in right hand.
- `mf-costume-persona-{color}`: a CSS `filter: hue-rotate(Xdeg)` on the character
  `<g>` for persona identification tinting.

**PixelMoufida new walk states** (extend `pixelArt/moufida.ts` drawChar):
- `"walk_left"` / `"walk_right"` — mirrored walk cycle for the companion window
  traversal animation (character walks across the companion canvas to a new position
  when changing context).
- `"jump"` — 2-frame hop used on celebrating state.

**Companion window traversal:**
Currently the companion shows the character static. Add a traversal animation:
when the companion window is repositioned or the state changes dramatically, the
character "walks" across the canvas to a new x-position at 2px/frame then stops and
plays the target state. Implemented in `companion.ts`'s `rAF` loop.

### H5.2 — Animation system for React components

A global CSS animation library (`frontend/src/styles/animations.css`) with reusable
keyframes imported into `index.css`:

```css
/* Entrance animations — applied with Tailwind-style utility classes */
@keyframes slide-up     { from { opacity:0; transform:translateY(16px) } to { opacity:1; transform:none } }
@keyframes slide-right  { from { opacity:0; transform:translateX(-16px)} to { opacity:1; transform:none } }
@keyframes fade-in      { from { opacity:0 }                             to { opacity:1 } }
@keyframes scale-in     { from { opacity:0; transform:scale(0.92) }      to { opacity:1; transform:scale(1) } }
@keyframes counter-up   { /* handled in JS via requestAnimationFrame */  }

/* Dashboard card entrance — staggered via CSS custom property */
.anim-card { animation: slide-up 0.35s ease both; animation-delay: calc(var(--i,0)*60ms) }

/* Score number count-up */
.anim-count { animation: counter-up 0.8s ease-out }

/* Notification toast slide-in */
.anim-toast { animation: slide-right 0.28s cubic-bezier(0.22,1,0.36,1) both }

/* Pulse ring for live indicators */
@keyframes pulse-ring {
  0%   { box-shadow: 0 0 0 0 rgba(var(--accent-rgb),0.4) }

  100% { box-shadow: 0 0 0 0 rgba(var(--accent-rgb),0) }

```

### H5.4 — Micro-interactions and decorations

**Button interactions:**
- All primary buttons: `transform: scale(0.97)` on `active`, eased back on release.
- Destructive buttons (delete, dismiss): brief red background pulse before action.
- "Approve" button on axis results: green ripple from click point.

**Card depth system:**
- Cards use a 3-level shadow system: `--shadow-1` (subtle, default), `--shadow-2`
  (hover lift), `--shadow-3` (focused/expanded).
- Hover lifts cards 2px with `transform: translateY(-2px)` + shadow escalation (200ms).

**Sidebar daemon status pill:**
- When daemon is `alive`: a green dot with `pulse-ring` animation.
- When paused: the dot turns amber with no pulse.
- When offline: grey, no animation.
- Transition between states: 300ms colour cross-fade.

**Event Feed timestamp:**
- Relative time ("2 min ago") counts up live via `setInterval` — no stale "X minutes ago"
  after long sessions.

**Score delta badges:**
- When a score changes (after a re-run triggered by daemon signal), the score number
  animates from old to new value using a count-up/count-down rAF loop.
- A delta badge (`+0.3` in green or `−0.4` in red) fades in beside the score, then
  fades out after 4s.

**Companion window improvements:**
- Add a subtle drop-shadow to the companion canvas so it sits above the desktop
  wallpaper more naturally.
- On hover: the character does a single quick wave (trigger `playWave()` from
  `PixelMoufida` on `mouseenter`).
- Right-click on companion: shows a small context menu — "Focus project", "Pause
  daemon", "Open Moufida" — without needing to open the main window.

**Background decorative elements:**
- A very low-opacity (3–5%) geometric pattern in the main content area background —
  a simple repeating diamond or hex grid in the accent colour. Subtle enough to add
  texture without distraction.
- The landing page gets a slow-moving radial gradient background (CSS `@keyframes`
  shifting hue slightly) to feel alive on first impression.

### H5.5 — Loading and empty states

**Skeleton screens:**
Every card that fetches data shows a correctly-sized shimmer skeleton before content
loads — not a generic spinner. The skeleton matches the shape of the loaded content
(e.g., a 3-row skeleton for CompetitorBoard, a ring skeleton for ScoreGauge).

**Empty states with character:**
- Competitor board empty: character in `"pointing_left"` state with speech bubble
  "No competitor data yet — the daemon is watching."
- Opportunity radar empty: character in `"idle"` with "No grants found in your sector
  yet. Check back soon."
- Event feed empty: character `"sleeping"` with "Nothing to report — focus a project
  to start watching."
- These replace the current plain text empty states.

**Error states:**
- When an API call fails, a toast notification slides in from the right with a red
  accent and an error icon.
- The character briefly shows `"worried"` state, returns to `"idle"` after 2s.

## Key shared component: `EvidenceTrace`

All three analytical features (H1–H3) share a single `EvidenceTrace` component that
renders the source chain for any claim. This is the grounding contract made visible:

```tsx
<EvidenceTrace refs={[
  { label: "market axis", field: "validated_customers", value: "null" },
  { label: "KB", doc: "ANPR_2024_agrifood_report.pdf", section: "§3.2" },
  { label: "competitor snapshot", name: "AgroManager TN", field: "pricing" }
]} />
```

Renders as a collapsible panel with colour-coded source type badges (axis = blue,
KB = purple, competitor = orange, daemon signal = green). Clicking a KB source opens
a modal with the retrieved chunk text. Clicking an axis source navigates to that axis
card on the dashboard.

This component alone communicates to judges that every output is traceable —
not a hallucination.
